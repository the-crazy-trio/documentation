# Product Proposal — AI Financial Assistant

## 1. Обоснование идеи

Частные инвесторы регулярно принимают решения на основе разрозненных источников информации:

- брокерские приложения показывают только позиции и цену;
- макроэкономические показатели и отчёты регуляторов находятся в отдельных источниках;
- для анализа динамики активов и сопоставления их с макрофинансовыми показателями требуется ручная работа.

В результате инвестор сталкивается с несколькими проблемами:

- сложно быстро понять текущее состояние портфеля;
- сложно сопоставить движение цены актива с макроэкономическим контекстом;
- проверка инвестиционной гипотезы требует поиска данных, построения графиков и интерпретации показателей.

Идея проекта — создать AI-ассистента, который:

- получает данные портфеля из брокера;
- агрегирует финансовые показатели из доверенных источников;
- анализирует динамику цен активов;
- предоставляет пользователю аналитическую сводку и визуализации по запросу.

---

# 2. Цель и метрики

## Продуктовые метрики (PoC)

### Корректность оценки соответствия портфеля стратегии (Strategy-fit correctness)
Насколько верно ассистент выдает:
- вердикт “соответствует/не соответствует/частично”
- 1–2 ключевых причины

- Метрика: accuracy вердикта + top-2 match причин
- Target PoC: ≥60% на небольшом экспертно размеченном сете

---

## Агентские метрики (PoC)

### Корректность интерпретации стратегии (Strategy parsing accuracy)
Насколько правильно агент извлекает параметры стратегии (слоты).

- Метрика: slot-level accuracy / F1
- Target PoC: ≥75% на небольшом валид-сете

### Привязка выводов к данным (Groundedness)
Доля ответов, где выводы опираются на реальные рассчитанные значения (доли, доходность, P/L, волатильность и т.д.).

- Метрика: pass/fail по чек-листу groundedness
- Target PoC: ≥80%

### Доля “придуманных” фактов (Fabrication rate)
Доля ответов с числами/фактами, которых нет в данных.

- Метрика: % ответов с fabrication
- Target PoC: ≤10%

---

## Технические метрики (PoC)

### Успешность запросов
Доля запросов, завершившихся ответом (включая graceful fallback при проблемах источников).

- Target PoC: ≥90%

### Latency (фиксируем ожидание, не оптимизируем в PoC)
- p95 end-to-end latency: ≤30 секунд для “тяжелого” запроса (с графиками)


# 3. Потенциальные сценарии использования

## Сценарий 1 — Подключить брокера и получить “снимок портфеля”
Пользователь подключает брокера (read-only) и просит краткий разбор текущего состояния.

Запросы:
- «Подтяни мой портфель и дай сводку»
- «Какие топ-позиции и где концентрация?»

Выдача:
- текущие позиции и доли (asset allocation)
- концентрация (топ-1/топ-5 по долям)
- прибыль/убыток и доходность — если брокер отдает нужные данные (cost basis / операции)

Edge-кейсы:
- нет cost basis / истории операций → выводим только структуру и динамику цен, P/L считаем “ограниченно” и явно отмечаем это
- часть активов не маппится на тикер/ISIN → помечаем “unknown”, не используем в аналитике, показываем список “что не распознано”

---

## Сценарий 2 — Пользователь описывает стратегию → оценка “портфель vs стратегия”
Пользователь задает стратегию свободным текстом или структурой. Ассистент извлекает параметры стратегии и проверяет портфель по доступным метрикам.

Запросы:
- «Стратегия: умеренный риск, долгосрок, диверсификация, меньше техов. Оцени мой портфель»
- «Стратегия: 60/30/10 акции/облигации/кэш, валютная диверсификация. Насколько соответствует?»

Выдача:
- интерпретация стратегии в виде 5–10 слотов (горизонт, риск, классы активов, ограничения)
- вердикт: соответствует / частично / не соответствует
- 2–3 причины, каждая с числовой опорой (доли, концентрация, валюты, класс активов)
- “что поправить” в терминах направлений (увеличить/уменьшить долю классов/секторов), без конкретных сделок

Edge-кейсы:
- стратегия слишком общая → 1 уточняющий вопрос (или фиксируем допущение и помечаем его)
- стратегия противоречива → показываем конфликт и почему по данным это конфликт
- стратегия требует метрик, которых нет в PoC (ESG, див. доходность, кредитные рейтинги) → явно сообщаем, что критерий не поддерживается и не используем его в вердикте

---

## Сценарий 3 — Рекомендация по диверсификации как “план выравнивания”
Пользователь просит: “как диверсифицировать”, но в PoC это делается простыми правилами и измеримыми сигналами, без сложных моделей.

Запросы:
- «Как улучшить диверсификацию под мою стратегию?»
- «Где у меня перекос и почему это риск?»

Выдача:
- список перекосов (концентрация в позиции/секторе/валюте/классе активов), с числами
- целевые диапазоны (либо из стратегии, либо дефолтные профили PoC: консервативный/умеренный/агрессивный)
- 3–5 шагов “уменьшить/увеличить долю X до диапазона Y–Z” (без тикеров)

Edge-кейсы:
- нет классификации по секторам/классам для части активов → рекомендации строятся по покрытой части, остальное помечается как “unknown” и снижает уверенность

---

## Сценарий 4 — Вопросы к портфелю в свободной форме (text2sql-стиль)
Пользователь задает короткие вопросы, которые сводятся к фильтрам/агрегациям по портфелю.

Запросы:
- «Какие позиции больше 5%?»
- «Сколько у меня USD-экспозиция?»
- «Топ-3 позиции по доле»

Выдача:
- таблица/список + короткая интерпретация “что это означает для портфеля”

Edge-кейсы:
- неоднозначные термины (“прибыль” без периода/метода) → PoC задает уточнение или использует дефолт и явно его пишет
- нет данных для конкретного запроса → частичный ответ + явное перечисление, чего не хватает

# 4. Ограничения

## Технические

**SLO / надежность (end-to-end)**
- Успешные ответы (полный ответ либо корректный partial/fallback с объяснением): ≥90%

**Время ответа (делаем мягко)**
- p95 end-to-end для “тяжелого” запроса (портфель + оценка по стратегии): ≤7–8 минут
- “Легкие” запросы (простые выборки/агрегации по портфелю): целевой ориентир ≤1–2 минуты (не строгий SLO)

**Актуальность данных**
- Котировки: допускается задержка до 15–60 минут (в ответе всегда показывать timestamp данных)
- Макро/секторные индикаторы: по последней доступной публикации (в ответе показывать дату источника)

**Ограничения брокерских данных**
- Только read-only, без сделок
- P/L и “прибыльность” возможны только при наличии cost basis/истории операций у брокера; иначе — ограниченная аналитика (структура, доли, динамика цен)
- Неполный маппинг инструментов (тикер/ISIN → котировки/класс/сектор) допустим; “unknown” активы явно показываются и частично исключаются из оценок

---

## Операционные

**Бюджет: стоимость LLM на 1 запрос (оценка через OpenRouter)**
OpenRouter биллит по input/output токенам по ставкам модели, без наценки; при routing/fallback списание идет только за успешный прогон модели.

**Допущение по токенам (тяжелый запрос)**
- Вариант A (1 вызов): ~12k input токенов + ~2k output токенов
- Вариант B (2 вызова: парсинг стратегии → итоговый разбор): ~15k input + ~2.5k output

**Примерные цены моделей на OpenRouter**
- openai/gpt-4o-mini: $0.15/M input, $0.60/M output
- google/gemini-2.5-flash: $0.30/M input, $2.50/M output
- anthropic/claude-3.5-sonnet: $6/M input, $30/M output

**Оценка стоимости 1 “тяжелого” запроса**
- openai/gpt-4o-mini:
  - A: ~$0.003 за запрос
  - B: ~$0.0038 за запрос
- google/gemini-2.5-flash:
  - A: ~$0.0086 за запрос
  - B: ~$0.0108 за запрос
- anthropic/claude-3.5-sonnet:
  - A: ~$0.132 за запрос
  - B: ~$0.165 за запрос

# 5. Архитектурный набросок

**Описание компонентов:**

## 1 Telegram UI
**Telegram Bot**

Основная точка входа: команды/кнопки, возврат картинок, флоу авторизации, подключения апи брокера

---

## 2. Backend (монолит) — базовые компоненты

**2.1 API слой**
- Обрабатывает запросы от бота.
- Управляет сессиями, доступом к данным пользователя.

**2.2 Router / Orchestrator**
- Определяет “что пользователь хочет” и выбирает специализированного агента.
- Передает агентам общий контекст: стратегия пользователя, снапшот портфеля, доступные источники данных, ограничения PoC.
- Собирает итоговый ответ из результатов агентов.

---

## 3. Agents Manager

Управляет в себе агентами, которые вызываются по api или как отдельные процессы

Первоначальные зоны ответственности под каждого агента:

**3.1 Strategy Agent (интерпретация стратегии)**

Преобразует стратегию, описанную в свободной форме в специализированный формат, взаимодействует с памятью

**3.2 Portfolio Analysis Agent (оценка портфеля/актива по стратегии)**

Использует стратегию и аналитические данные согласно активам портфеля для выдачи аналитики пользователю по запросу

**3.3 Text2SQL Agent (вопросы к данным портфеля)**

Используется в тандеме с визуализатором - помогает выделить нужные данные и привести в удобный вид для отрисовки

**3.4 Dashboard/Visualization Agent (генерация дашбордов/графиков)**

Работает с таблицами данных, умеет отрисовывать необходимые графики, диаграммы, дашборды. Отдает все в формате png

---

## 4. Storages

### 4.1. User Storage

БД для хранения служебной информации - типа данные сессий, айдишники чатов, пользователей, их ключи, данные от аккаунта брокера

### 4.2. Analytics Storage

БД (типа кликхаус) для хранения макро и микро экономических показателей активов, компаний и т.д.

### 4.3. Memory Storage

БД для хранения памяти агентов (сюда сгружаются стратегии, предпочтения)

## 5. Integrations

### 5.1. Openrouter

Инференс LLM по API

### 5.2. T-Invest API

Для работы с портфелем

### 5.3. Python/TS Sandbox API

Для отрисовки графиков

Система:

## Шаг 5 — Запрос базовой аналитики портфеля
Пользователь запрашивает: “Покажи аналитику портфеля”.

Система:
1) Подтягивает актуальный снапшот портфеля (из памяти или повторно из T-Invest API при необходимости)
2) Обогащает данные:
   - маппинг инструментов (FIGI/ISIN/тикер)
   - ретрив цен и истории из **Analytics Storage** (и/или market data источника)
3) Считает детерминированные метрики портфеля:
   - аллокации, концентрация, вклад позиций
   - P/L и доходность (если доступны нужные поля)
   - базовые риск-прокси (если есть история)
4) Сохраняет результат как `AnalyticsSnapshot` в **Memory Storage** (для последующих шагов)

Agent: **да**

## Шаг 6 — Запрос специфичного графика
Пользователь просит: “Сделай график X” (например, вклад топ-5 позиций в доходность за год).

Система:
- формирует выборку данных (из `AnalyticsSnapshot` или через запрос к **Analytics Storage**)
- подготавливает таблицу для отрисовки
- передает данные в **Python/TS Sandbox API**
- получает PNG и сохраняет артефакт (id/ссылка) в **User Storage** или отдельной таблице артефактов
- возвращает PNG пользователю (в Mini App или сообщением в боте)

Agent: **да**
---

## Шаг 7 — Запрос показателей одного конкретного актива
Пользователь выбирает актив и спрашивает: “Покажи показатели по активу A”.

Система:
- определяет идентификатор инструмента (FIGI/ISIN/тикер)
- запрашивает из **Analytics Storage** (и/или market data источника):
  - текущую цену
  - историю за период
  - базовые метрики (например, изменение, волатильность-прокси, просадка — если считаем)
- возвращает пользователю сводку по активу (и при необходимости отдельный график через Sandbox)

Agent: **да**
---

## Шаг 8 — Аналитика портфеля по стратегии
Пользователь просит: “Оцени портфель по моей стратегии”.

Система:
1) Загружает стратегию из **Memory Storage** (слоты)
2) Загружает последний **`AnalyticsSnapshot`** (или пересчитывает, если данные устарели)
3) Выполняет проверки соответствия стратегии:
   - доли классов/валют/секторов относительно требований
   - концентрация относительно ограничений
   - прочие доступные критерии (только те, под которые есть данные)
4) Формирует итоговый результат:
   - вердикт (соответствует/частично/нет)
   - 2–3 причины, каждая с числовой опорой из snapshot
   - “как выровнять” в терминах направлений (изменить доли/экспозиции), без конкретных сделок
5) Сохраняет результат оценки (версия стратегии + timestamp + краткая сводка) в **Memory Storage** (и/или в историю оценок)

Agent: **да**

  - ссылки на использованные метрики/снапшоты
  - (опционально) прикрепленные PNG-графики
