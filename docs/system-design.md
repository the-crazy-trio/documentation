# Дизайн PoC-системы

## Назначение

Этот документ фиксирует архитектуру PoC-системы `Financial AI Assistant` перед реализацией.

Документ покрывает:
- состав системы и зоны ответственности модулей;
- основной сценарий исполнения;
- модель взаимодействия с LLM;
- контракты инструментов и внешних интеграций;
- guardrails, fallback и контроль качества;
- технические и операционные ограничения;
- недостающие артефакты, без которых реализация будет слишком неопределенной.

Дизайн намеренно ориентирован на прагматичный PoC:
- детерминированные финансовые расчеты важнее агентской автономности;
- доступ к инструментам строго ограничен оркестратором;
- лучше частичный ответ с явными ограничениями, чем правдоподобная, но недостоверная генерация;
- архитектура должна быть достаточно конкретной, чтобы перейти к реализации без крупных пробелов.

## Цели

- Поддержать импорт портфеля из брокера в режиме read-only.
- Поддержать Q&A по портфелю, парсинг стратегии, strategy-fit-анализ и простые рекомендации по диверсификации.
- Поддержать рекомендации по портфелю на основе стратегии пользователя в виде аналитических направлений, а не торговых сигналов.
- Поддержать построение простых графиков и PNG-артефактов.
- Гарантировать, что все финансовые выводы в ответе опираются на полученные или вычисленные данные.
- Встроить измеримый контроль качества, latency, cost и fallback-поведения.

## Не-цели

- Автоматическая торговля и исполнение сделок.
- Полноценный анализ новостей как обязательный контур PoC.
- Сложные valuation-модели, прогнозирование, backtesting.
- Production-grade масштабирование, комплаенс и долгосрочное хранение персональных данных.
- Полностью автономные агенты с произвольным выбором инструментов.

## Ключевые архитектурные решения

### 1. Один backend-оркестратор

PoC реализуется как единая backend-система с центральным оркестратором.

"Агенты" в этом дизайне — это не независимые сервисы со свободой действий, а режимы исполнения внутри оркестратора. Оркестратор:
- классифицирует запрос;
- собирает контекст;
- выбирает разрешенные шаги исполнения;
- вызывает только разрешенные инструменты;
- валидирует промежуточные и финальные результаты;
- решает, когда вернуть полный ответ, частичный ответ или безопасный отказ.

Такой подход упрощает контроль качества, безопасность и observability на этапе PoC.

### 2. Deterministic-first analytics

Все числа, которые пользователь видит в ответе, должны происходить из:
- данных брокера;
- market/macro источников;
- детерминированного вычислительного контура.

LLM может:
- интерпретировать результат;
- ранжировать причины;
- задавать уточняющий вопрос;
- формулировать финальный ответ.

LLM не должен:
- самостоятельно считать финансовые метрики;
- подставлять отсутствующие данные;
- домысливать unsupported criteria;
- формировать числа, которых нет в evidence bundle.

### 3. Структурированная память важнее сырого текста

Стратегия пользователя один раз нормализуется в structured schema и дальше используется как каноническое представление.

Сырой текст стратегии может храниться для аудита и отладки, но downstream-модули должны работать со слотами и нормализованными ограничениями, а не с исходным свободным текстом.

### 4. Retrieval — это typed access, а не generic RAG

Retrieval-контур PoC строится вокруг ограниченного набора известных источников:
- брокерские данные по портфелю;
- market/macro данные;
- вычисленные снимки аналитики (`AnalyticsSnapshot`);
- память о стратегии и предыдущих оценках;
- артефакты визуализации.

Это не generic document-RAG. Основной retrieval в PoC — это source-aware и schema-aware доступ к данным через API, хранилища и специализированные retrieval tools.

### 5. Guardrails применяются не только prompt-ом, но и слоем исполнения

Prompt-level правила недостаточны. Безопасность и качество должны обеспечиваться за счет архитектуры исполнения:
- read-only доступ к брокеру;
- allowlist инструментов;
- guarded text2sql/text2api;
- sandbox-only рендеринг;
- redaction секретов и ограниченное логирование;
- валидатор финального ответа перед отправкой ответа пользователю.

## Обзор системы

### Основные модули

1. `Telegram Bot / Mini App`
Точка входа для пользователя: команды, сообщения, подтверждения, возврат текста и изображений.

2. `API Layer`
Принимает запросы от клиента, валидирует пользователя и сессию, нормализует входной payload и передает его оркестратору.

3. `Orchestrator`
Центральный контроллер системы. Управляет маршрутизацией запроса, контекстом, вызовами LLM, retrieval-путями, fallback-логикой и финальным ответом.

4. `Strategy Parser`
LLM + schema validation для преобразования свободного текста стратегии в структурированную модель.

5. `Portfolio Analytics Engine`
Детерминированный расчет аллокаций, концентрации, экспозиций, доходности при наличии данных, простых risk proxies и strategy-fit checks.

6. `Text2SQL / Text2API / Structured Query Path`
Контур для вопросов к данным портфеля. LLM может построить query plan, SQL или нормализованный API intent, но фактическое исполнение происходит только через guarded retrieval layer.

7. `Visualization Service`
Готовит датасет для визуализации и передает его в sandbox-renderer для генерации PNG.

8. `Retriever / Tool Layer`
Единый слой доступа к T-Invest API, market/macro источникам, memory storage, analytics storage и guarded data-query инструментам.

9. `User Storage`
Хранит идентификаторы пользователя, сессии, состояние подключения брокера, ссылки на артефакты и служебные пользовательские данные.

10. `Memory Storage`
Хранит стратегию, снимки портфеля (`PortfolioSnapshot`), снимки аналитики (`AnalyticsSnapshot`), краткие summary прошлых оценок и служебные derived states.

11. `Analytics Storage`
Хранит market/macro временные ряды, маппинг инструментов, справочники, а также данные для повторного расчета и визуализаций.

12. `Observability / Evals`
Собирает технические метрики, логи, трассировки и offline evaluation results.
На раннем PoC этапе может быть реализован частично, но схема событий и метрик должна быть заложена сразу.

## Роли LLM в системе

LLM используется только для ограниченных задач:
- классификация пользовательского intent;
- парсинг стратегии в structured schema;
- преобразование свободного вопроса в query plan для guarded retrieval;
- синтез финального grounded-ответа по уже валидированным данным;
- формулировка понятного частичного ответа или явного ограничения.

LLM не отвечает за:
- прямой доступ к инструментам;
- вычисление финансовых метрик;
- принятие решений о правах доступа;
- генерацию произвольного кода;
- вывод фактов вне доступного evidence bundle.

## Классы запросов

Оркестратор маршрутизирует запрос в один из поддерживаемых классов:

1. Подключение брокера и импорт портфеля
2. Снимок портфеля и описательная сводка
3. Аналитика по отдельному активу и график
4. Сохранение или обновление стратегии
5. Оценка портфеля относительно стратегии
6. Рекомендации по диверсификации
7. Q&A по данным портфеля в стиле `text2sql` или `text2api`
8. Запрос на визуализацию по готовому или вновь построенному датасету

Если запрос не попадает ни в один класс, система возвращает явный out-of-scope ответ.

## Основной workflow исполнения

### Шаг 1. Intake

API-слой валидирует пользователя, session context и входной payload. Запрос получает correlation id для трассировки.

### Шаг 2. Классификация intent

Оркестратор определяет класс запроса.
Для очевидных команд возможен rule-based путь.
Для свободного текста разрешен bounded LLM call.

### Шаг 3. Сбор контекста

Оркестратор загружает только релевантное состояние:
- `user_id`, `session_id`, `chat_id`;
- последнюю валидную структурированную стратегию (`StructuredStrategy`);
- последний снимок портфеля (`PortfolioSnapshot`);
- снимок аналитики (`AnalyticsSnapshot`), если он достаточно свежий;
- capability flags по доступным источникам.

### Шаг 4. Исполнение

В зависимости от класса запроса оркестратор может:
- запросить свежие брокерские данные;
- получить market/macro данные;
- пересчитать `AnalyticsSnapshot`;
- запустить strategy parser;
- запустить guarded text2sql/text2api;
- подготовить датасет для визуализации;
- вызвать sandbox-renderer.

### Шаг 5. Валидация

Перед финальным ответом оркестратор проверяет:
- все числовые утверждения опираются на evidence;
- unsupported criteria явно помечены;
- stale data раскрыта пользователю;
- не произошло cross-user leakage;
- tool outputs соответствуют schema, row limits и timeout budget.

### Шаг 6. Response synthesis

LLM получает только компактный validated evidence bundle и инструкции по безопасному grounded synthesis.

Финальный ответ обязан включать:
- краткий вывод;
- ключевые причины;
- timestamps / source dates, если это важно для интерпретации;
- caveats и unknown/unsupported зоны покрытия.

### Шаг 7. Сохранение результатов

Сохраняются только те сущности, которые нужны для повторного использования, дебага или истории:
- структурированную стратегию (`StructuredStrategy`);
- снимок портфеля (`PortfolioSnapshot`);
- снимок аналитики (`AnalyticsSnapshot`);
- assessment summary;
- visualization artifact refs.

## State / Memory / Context handling

### Session state

Короткоживущий state текущего диалога:
- `user_id`
- `chat_id`
- `session_id`
- активный request class
- флаги ожидаемого подтверждения
- correlation id / request id

### Long-lived memory

Долгоживущая пользовательская память:
- broker connection metadata
- strategy slots и версия схемы
- история версий стратегии
- последние снимки портфеля (`PortfolioSnapshot`)
- последние снимки аналитики (`AnalyticsSnapshot`)
- summary прошлых strategy-fit оценок
- ссылки на PNG-артефакты

### Политика памяти

- Structured strategy живет до замены или удаления.
- Сырой текст стратегии может храниться отдельно и с более строгим контролем доступа.
- `PortfolioSnapshot` и `AnalyticsSnapshot` имеют TTL и freshness metadata.
- Частичные результаты хранятся только с явным флагом `partial=true`.
- Старые snapshots не являются source of truth и могут удаляться.

### Политика контекста для LLM

Оркестратор должен использовать компактный structured context.

Всегда включать, если релевантно:
- request class;
- цель пользователя;
- актуальные strategy slots;
- минимальный набор аналитических фактов;
- unsupported criteria list;
- указание на freshness и coverage.

Включать по summary или ссылке:
- длинные таблицы портфеля;
- временные ряды;
- raw broker payloads;
- сырой текст стратегии.

Никогда не включать:
- секреты и токены;
- raw SQL без валидации;
- сырые логи;
- данные другого пользователя.

## Retrieval-контур

Retrieval разделен на typed paths.

### 1. Portfolio retrieval

Источник:
- T-Invest API;
- кэшированные snapshots портфеля.

Возвращает:
- позиции;
- количество;
- текущую стоимость;
- при наличии — операции и cost basis.

### 2. Market / reference retrieval

Источник:
- analytics storage;
- внешние market/macro providers.

Возвращает:
- mapping инструментов;
- цены и историю цен;
- классификации активов;
- макроиндикаторы и даты публикации.

Для текущего PoC mapping должен в первую очередь покрывать российский рынок и частично китайские инструменты, доступные через T-Invest API. `Country` не используется как обязательная аналитическая dimension и не участвует в verdict strategy-fit-анализа.

### 3. Memory retrieval

Источник:
- memory storage.

Возвращает:
- strategy slots;
- assessment summaries;
- переиспользуемые снимки аналитики (`AnalyticsSnapshot`).

### 4. Analytical retrieval

Источник:
- хранилище снимков аналитики (`AnalyticsSnapshot`).

Возвращает:
- детерминированно вычисленные метрики;
- концентрацию;
- экспозиции;
- coverage notes;
- unsupported / unknown portions.

### Правила retrieval

- Сначала использовать свежий валидный кэш, если он покрывает запрос.
- При устаревании данных пересчитывать analytics.
- Unknown/unmapped assets всегда помечать явно.
- Отсутствующие данные не должны компенсироваться догадкой LLM.

## Интеграции и tool/API contracts

### OpenRouter / LLM provider

Назначение:
- intent parsing;
- strategy parsing;
- grounded synthesis;
- bounded query planning.

Контрольные точки:
- выбор модели по request class;
- token caps;
- timeout caps;
- retry только на transient failures;
- запрет на попадание секретов в prompt и логи.

### T-Invest API

Назначение:
- получение портфеля, счетов и операций в read-only режиме.

Ограничения:
- только read-only доступ;
- timeout + retry с backoff;
- account ownership validation на каждом обращении;
- отсутствие write side effects.

Подробный рабочий контракт вынесен в `docs/specs/tinvest-api-contract.md`.

### Market / macro providers

Назначение:
- цены, история, классификации, макропоказатели.

Ограничения:
- обязательный timestamp/source date;
- stale data допустима только с disclosure;
- при недоступности источника система должна уметь вернуть partial answer.

### Sandbox renderer

Назначение:
- построение PNG-графиков и простых дашбордов.

Ограничения:
- только sandbox;
- без открытого доступа в сеть;
- без произвольного доступа к файловой системе;
- с лимитами CPU/RAM/time;
- на вход принимает только typed dataset и chart spec.

### Guarded retrieval tool

Назначение:
- безопасное исполнение запросов к данным портфеля.

Ограничения:
- allowlist views / tables / API actions;
- запрет DDL/DML;
- обязательный `user_id` filter;
- row limit и execution-time limit.

Подробные input/output schemas и error formats вынесены в `docs/specs/tool-schemas.md`.

## Failure modes, fallback и guardrails

### Неполные данные по портфелю

Если у брокера нет cost basis или часть активов не маппится:
- ответ строится по покрытой части портфеля;
- неподдержанная часть явно помечается;
- confidence / coverage снижается и отражается в ответе.

### Неопределенность стратегии

Если стратегия слишком общая, противоречивая или требует неподдерживаемых критериев:
- система задает один уточняющий вопрос, если это действительно повышает точность;
- либо фиксирует допущение;
- либо помечает критерий как unsupported и не использует его в вердикте.

Для текущего PoC критерии по `country` следует считать unsupported, если пользователь явно не сформулировал их как информационный, а не блокирующий аналитический сигнал.

### Сбой внешнего API

Если T-Invest API или market source недоступен:
- выполняются ограниченные retries;
- затем используется последний допустимый snapshot, если он есть и свежесть приемлема;
- иначе возвращается partial/unavailable response с объяснением.

### Защита от hallucination

Ответ не должен содержать числа, отсутствующие в evidence bundle.
Если validator не находит подтверждение для числового утверждения, черновик ответа отклоняется.

### Защита от injection

Пользовательский текст не меняет tool permissions.
Strategy parser нормализует ввод до structured slots.
SQL/API вызовы и рендеринг возможны только через guarded system components.

### Подтверждение чувствительных действий

Хотя PoC не поддерживает сделки, ряд действий должен подтверждаться явно:
- отключение брокера;
- удаление стратегии;
- удаление пользовательских данных;
- сброс памяти.

## Контроль качества

Контроль качества сосредоточен на поведении LLM и корректности evidence chain.

### Основные метрики

- `Strategy parsing accuracy`
- `Groundedness`
- `Fabrication rate`
- `Strategy-fit correctness`
- `Fallback success rate`
- `Unsupported-criteria disclosure rate`

### Online checks

- schema validation на strategy slots и tool outputs;
- validator числовых утверждений в финальном ответе;
- проверка явного disclosure unsupported criteria;
- проверка freshness disclosure;
- telemetry по retries, timeouts и cost.

### Offline evals

Подробные benchmark-целевые значения и состав eval-сетов вынесены в `docs/specs/offline-eval-benchmarks.md`.

## Технические и операционные ограничения

### Reliability

- Доля успешных ответов, включая корректные частичные ответы: не ниже `90%`.

### Latency

В текущих документах есть конфликтующие оценки latency. В этом дизайне они нормализованы по классам запросов:
- легкие запросы: целевой ориентир `<= 1-2 минуты`;
- тяжелые запросы с обновлением портфеля, аналитикой и графиками: мягкий ориентир `<= 7-8 минут`.

Значение `<= 30s` можно рассматривать как demo-target для ограниченного happy path, но не как общий бюджет тяжелого сценария.

### Cost

- минимизировать число LLM-вызовов на request path;
- кэшировать стратегию и снимки аналитики (`AnalyticsSnapshot`);
- использовать дешевые модели для классификации и парсинга, если они проходят quality gates;
- использовать более дорогую модель для synthesis только при необходимости.

### Data freshness

- котировки могут отставать на `15-60 минут`, если это явно раскрыто;
- macro данные используются по последней доступной публикации;
- важные ответы всегда должны показывать timestamps/source dates.

## Контрольные точки перед реализацией

До старта реализации должны быть зафиксированы и утверждены:

1. Контракт request classification и полный список request classes.
2. Structured strategy schema и правила валидации.
3. Analytics snapshot schema и freshness policy.
4. Guarded text2sql/text2api contract и allowlisted data surface.
5. Sandbox visualization contract.
6. Validator rules для groundedness и unsupported criteria.
7. Logging/redaction policy.
8. Контракт адаптера T-Invest API.
9. Tool I/O schemas и унифицированные error messages.
10. Offline evaluation benchmarks и gold datasets.

## Открытые решения

- Нужно ли со временем мигрировать `Analytics Storage` с `SQLite` на `Postgres` или `ClickHouse` при росте объема данных.
- Нужно ли вводить отдельный query-plan DSL после PoC, если `SQL/API intent schema` окажется недостаточно выразительной или безопасной.
- Нужен ли полноценный `Mini App` как first-class интерфейс после bot-first этапа.
- Нужно ли расширять observability-контур сверх базовых трассировок в `Langfuse`.

## Зафиксированные уточнения

- Для `User Storage` и `Memory Storage` допустимо использовать `MongoDB` на этапе PoC.
- Для `Analytics Storage` допустимо использовать `SQLite` на этапе PoC.
- Для query path достаточно `SQL/API intent schema`, отдельный DSL на PoC не требуется.
- Основной пользовательский интерфейс PoC: `Telegram Bot`.
- Базовый tracing-контур PoC: `Langfuse`.
- `Country` не является актуальной аналитической dimension для текущего PoC и не участвует в verdict strategy-fit-анализа.
- Основа enrichment для инструментов: внутренняя `mapping table` на базе идентификаторов `instrument_uid` / `figi` из T-Invest API.

Эти решения считаются принятыми для текущей версии PoC и не требуют дополнительного согласования перед началом реализации.
